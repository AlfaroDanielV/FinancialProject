# CLAUDE.md ‚Äî Personal Finance Agent

## What This Project Is

A personal finance agent that automates transaction capture, parses bank emails, generates financial reports, and provides a conversational AI layer capable of giving financial advice ‚Äî including pushing back on unrealistic goals. Built as a personal tool first, with a clear path to multi-tenancy if commercialized.

**The core thesis**: If the ledger is wrong, the agent is useless. Data accuracy is a prerequisite to any AI layer. Every architectural decision flows from this.

---

## Long-Term Vision

1. **Personal MVP** ‚Äî A system I trust enough to check instead of my bank app
2. **Stabilization** ‚Äî 4+ weeks of reliable daily use with accurate weekly reports
3. **Product** ‚Äî Multi-tenant SaaS sold to individuals and couples via Telegram bot, API-backed LLM, no dashboard (chat-native UX)

The product is **not** a dashboard app. It's a financial assistant that lives in a messaging thread. No login portals, no charts page, no React frontend. The interface is the chat.

### Why NOT these alternatives

- **WhatsApp / Baileys**: Unofficial API, Meta bans accounts using it. Telegram Bot API is official and zero ban risk.
- **Self-hosted LLMs (Gemma 4, DeepSeek)**: GPU server starts at ~$80/mo. API costs are ~$5/mo at 10 users, ~$40/mo at 100. Self-host only makes sense at 10k+ daily active users.
- **LLM-generated pushback**: Financial pushback must be deterministic and rule-based. LLMs are for empathy and explanation, not for doing the math.
- **Autogenerated Alembic migrations**: Hand-written gives full control over schema changes.

## Database Schema (Core Tables)

All tables use UUID primary keys via `gen_random_uuid()`. Timestamps are `TIMESTAMPTZ`. Amounts are `NUMERIC(12,2)` ‚Äî negative = expense, positive = income.

- **users** ‚Äî `id`, `name`, `currency` (default CRC), `timezone` (default America/Costa_Rica)
- **accounts** ‚Äî `id`, `user_id` FK, `name` ("Chase Checking"), `account_type` (checking/savings/credit/investment), `is_active`
- **transactions** ‚Äî `id`, `user_id` FK, `account_id` FK, `amount`, `currency`, `merchant`, `description`, `category`, `transaction_date`, `source` (manual/email/import), `source_ref` (email Message-ID for dedup), `parse_status` (parsed/needs_review/failed), `raw_data` (JSONB, original email text), `tags` (ARRAY), `is_recurring`
- **budgets** ‚Äî `id`, `user_id` FK, `category`, `amount`, `period` (weekly/monthly), `start_date`, `is_active`
- **goals** ‚Äî `id`, `user_id` FK, `name`, `target_amount`, `current_amount`, `deadline`, `priority`, `status` (active/paused/completed/abandoned)
- **weekly_reports** ‚Äî `id`, `user_id` FK, `week_start`, `week_end`, `report_data` (JSONB), `sent_at`
- **debts** / **debt_payments** ‚Äî loan schedules (Phase 3); amortization engine in `api/services/amortization.py`.

### Phase 4 tables (recurring bills + calendar alerts)

All amounts in Phase 4 tables are `NUMERIC(14,2)`. Dates are calendar `DATE`; timestamps `TIMESTAMPTZ`.

- **recurring_bills** ‚Äî template for a recurring charge. `id`, `name`, `provider`, `category` (bill category enum), `amount_expected` (nullable if `is_variable_amount`), `currency`, `is_variable_amount`, `account_id` FK ‚Üí accounts, `frequency` (weekly/biweekly/monthly/bimonthly/quarterly/semiannual/annual/custom), `day_of_month` (1‚Äì31, CHECK), `recurrence_rule` (iCal RRULE, required when frequency=custom), `start_date`, `end_date`, `lead_time_days`, `is_active`, `notes`, `linked_loan_id` FK ‚Üí debts (nullable, `ON DELETE SET NULL`).
- **bill_occurrences** ‚Äî materialized instance. `id`, `recurring_bill_id` FK cascade, `due_date`, `amount_expected`, `amount_paid`, `status` (pending/paid/partially_paid/skipped/overdue/cancelled), `paid_at`, `transaction_id` FK ‚Üí transactions, `notes`. **UNIQUE(recurring_bill_id, due_date)** guarantees idempotent regeneration.
- **custom_events** ‚Äî one-off or recurring calendar events not tied to a bill. `id`, `title`, `description`, `event_type` (tax_deadline/goal_milestone/income_expected/reminder/other), `event_date`, `is_all_day`, `event_time`, `amount`, `currency`, `recurrence_rule`, `is_active`.
- **notification_rules** ‚Äî advance-notice configuration. `id`, `scope` (bill/event/category_default/global_default), `recurring_bill_id` / `custom_event_id` / `category` (mutually exclusive per scope, enforced by CHECK `ck_notification_rules_scope_target`), `advance_days` (JSONB list of ints, descending), `is_active`. Seed: one row with `scope=global_default` and `advance_days=[7,3,1,0]`.
- **notification_events** ‚Äî generated alerts ready for a channel worker to pick up. `id`, `bill_occurrence_id` xor `custom_event_id` (CHECK `ck_notification_events_target`), `trigger_date`, `advance_days`, `channel` (in_app/telegram/email), `status` (pending/delivered/acknowledged/dismissed/failed), `delivered_at`, `acknowledged_at`, `payload_snapshot` (JSONB ‚Äî frozen at creation so later edits to the source row don't corrupt the alert).

**Rule resolution order** for an occurrence or event: specific `scope=bill` / `scope=event` rule ‚Üí `scope=category_default` matching `recurring_bill.category` ‚Üí `scope=global_default`. First match wins.

**Idempotency contract**: `generate_occurrences`, `mark_overdue`, and `compute_pending_notifications` must be safe to run on any schedule. The first relies on `UNIQUE(recurring_bill_id, due_date)`; the third de-dups in code by `(target_kind, target_id, advance_days)`.

**Deduplication strategy**: `source_ref` holds the Gmail Message-ID. Before inserting an email-parsed transaction, check for existing rows with the same `source_ref` + `user_id`. Also fuzzy-match on `amount` + `merchant` + `transaction_date` within a ¬±1 day window for manual vs email duplicates.

## Phase 5b ‚Äî Telegram bot + LLM extraction

The bot lives in `bot/` (sibling of `api/`). The pipeline is strictly:

```
Telegram update
  ‚Üí resolve user (telegram_user_id ‚Üí users.id)
  ‚Üí [rate limit gate + daily token budget gate]
  ‚Üí command / confirmation short-circuit (no LLM if /help, /undo, /cancel, "s√≠", "no")
  ‚Üí LLM extractor (Anthropic, tool use) ‚Üí ExtractionResult (Pydantic)
  ‚Üí deterministic routing ‚Üí query dispatcher OR write Dispatcher
  ‚Üí write Dispatcher ‚Üí ProposeAction | AskClarification
                            | ConfirmResponse | UndoRequest | ShowHelp | Reject
  ‚Üí if dispatcher="query": run app.queries.dispatcher and send the reply
  ‚Üí if ProposeAction: stage PendingAction in Redis, send S√≠/No/Editar keyboard
  ‚Üí if AskClarification: ask question, store partial in Redis
```

**Hard rule (do not relax):** the LLM extracts, deterministic routing decides whether this is write/query/control. The write dispatcher never asks an LLM what to do. The query dispatcher is the explicit read-only LLM/tool-use path after routing has already selected `dispatcher="query"`.

### Library + model choices

- **aiogram v3** (not python-telegram-bot). Native asyncio, cleanest FastAPI integration, webhook is a one-liner (`dp.feed_update(bot, update)`). Built-in FSM is NOT used for durable state ‚Äî see below.
- **Claude Haiku 4.5** for extraction (`LLM_EXTRACTION_MODEL` env var). ~$0.40 / 1000 messages with prompt caching on. Sonnet swap is one env-var change if CR-slang fixtures reveal accuracy gaps.
- **Prompt caching on from day 1.** `cache_control=ephemeral` on both the system prompt and tool schema blocks. Uncached runs burn input tokens fast during fixture-test dev ‚Äî do not ship uncached.

### State storage

Redis is the source of truth for durable bot state. aiogram's FSM is permitted ONLY for transient in-handler dialog bookkeeping; anything that must survive a restart or a webhook-mode deploy goes in Redis. Key conventions in `bot/redis_keys.py`:

| Key | Purpose | TTL |
|---|---|---|
| `telegram:pairing:{code}` | Pairing code ‚Üí user_id | 300s |
| `telegram:pending:{user_id}` | Staged proposal awaiting S√≠/No/Editar | 300s |
| `telegram:last_action:{user_id}` | Last committed action id for /undo | 24h |
| `telegram:rate:{user_id}:{minute}` | Fixed-window rate counter (30/min) | 65s |
| `telegram:clarification:{user_id}` | Write-flow clarification partial | 300s |
| `query_history:{user_id}` | Text-only query conversation history | 24h |

Daily LLM budget is no longer a Redis counter. `api.services.budget`
sums `llm_extractions` + `llm_query_dispatches` since the user's local
midnight and raises `BudgetExceeded` before the next LLM call would cross
the cap.

### `llm_extractions` table (migration 0007)

Every LLM call writes one row: `user_id`, SHA-256 `message_hash` (not raw text ‚Äî personal finance stays out of INFO logs), `intent`, `confidence`, the full ExtractionResult JSON, latency, and token counts (including cache read / creation split). Purpose is evaluation (debugging "the bot misunderstood me" reports), not analytics. Indexed by `(user_id, created_at DESC)`.

### Extraction contract

`ExtractionResult` (Pydantic v2, `extra="forbid"`) fields: `intent`, `dispatcher`, `amount`, `currency`, `merchant`, `category_hint`, `account_hint`, `occurred_at_hint`, `query_window`, `confidence`, `raw_notes`. Validators:
- `dispatcher` is required and must be `write | query | control`. Queries use `intent="query"` plus `dispatcher="query"`; legacy `query_recent` and `query_balance` intents were removed in Phase 6a.
- `currency` normalized to `CRC|USD` or dropped to `None` (dispatcher then defaults to user.currency).
- `query_window` validates the enum + `last_n_days:<N>` shape.
- `amount` must be strictly positive (sign applied by dispatcher).
- Unknown / malformed values fall to `None` rather than raising ‚Äî the dispatcher prefers partial honesty to full hallucinations.

### Dispatcher pass-through rules

- `category_hint` ‚Üí `transactions.category` verbatim (whitespace-normalized). **No synonym map** ‚Äî the schema is free-form and premature normalization would hide real drift. Add mapping later with evidence from actual usage.
- `account_hint` ‚Üí rapidfuzz WRatio against `accounts.name`. Single active account auto-selects regardless of hint. Multiple + no clear match ‚Üí clarify.
- `occurred_at_hint` ‚Üí small Spanish table (hoy, ayer, anteayer). Anything else falls back to today; summary surfaces the resolved date so the user can correct via Editar.
- `confidence < 0.6` ‚Üí clarify for write intents. Query ambiguity is handled by `app.queries.dispatcher` inside the tool-use loop and conversation history; query turns do not enter the write clarification state.

### `/undo` semantics

Hard delete of the last committed transaction, guarded by three checks:

1. Row must exist and belong to the caller.
2. `transactions.source == 'telegram'` ‚Äî we refuse to delete rows created via manual REST, email parse, or iPhone Shortcut.
3. No `bill_occurrences.transaction_id` references it ‚Äî if the user already marked a bill paid with this transaction, /undo refuses and points them to un-marking the bill first.

Only the last action is undoable; deeper history requires the REST API.

### `_simulate` endpoint (dev only)

Two modes of the same endpoint:
- Real LLM path: POST `{telegram_user_id, text}` ‚Äî runs extractor + dispatcher + commit.
- Zero-cost mock path: POST `{telegram_user_id, text, mock_extraction: {...}}` ‚Äî skips the LLM and feeds the dict through `ExtractionResult.model_validate`. Used by `scripts/phase5b_smoke.sh` so CI doesn't burn Anthropic tokens.

Also accepts `pairing_code` to drive the /start pairing flow from curl without a real Telegram client.

### Testing

- `tests/test_llm_extractor.py` ‚Äî fixture tests with `FixtureLLMClient`. Deterministic, offline, pins the current `ExtractionResult` schema including required `dispatcher`.
- `tests/test_telegram_dispatcher.py` ‚Äî pure unit tests for the write dispatcher. No LLM, no DB. Query intents are asserted to be rejected here because routing sends them to `app.queries.dispatcher` before the write dispatcher.
- `scripts/phase5b_smoke.sh` ‚Äî end-to-end against `_simulate` (pair ‚Üí propose ‚Üí confirm ‚Üí undo ‚Üí balance ‚Üí help).

Re-record extractor fixtures against the real API when the prompt or model changes. Drift is a signal prompt-engineering needs attention; don't loosen the assertions.

## Phase 6a ‚Äî Conversational query layer

Phase 6a adds a read-only, tool-using query dispatcher for financial
questions. The old deterministic query-write branch was removed from the
bot pipeline; Telegram queries now go:

```
Telegram text
  ‚Üí LLM extractor returns dispatcher="query"
  ‚Üí app.queries.dispatcher.run_dispatch()
  ‚Üí Anthropic tool-use loop over app.queries.tools.*
  ‚Üí llm_query_dispatches audit row
  ‚Üí text-only query_history append on success
  ‚Üí sanitize_telegram_html ‚Üí split_for_telegram ‚Üí sequential Telegram sends
```

### Query dispatcher behavior

- `app.queries.dispatcher.handle()` remains the compatibility entrypoint
  that returns `str`. New callers use `run_dispatch()` to get
  `DispatchOutcome` with `dispatch_id`, iterations, tools, token counters,
  duration, and error category.
- `llm_query_dispatches` (migration 0009 plus cache metrics in 0010)
  records one row per query message. It stores message hash, iterations,
  input/output tokens, cache read/creation tokens, tools used, duration,
  final response length, and error text.
- `app.queries.history` persists text-only `{role, content}` turns in
  Redis under `query_history:{user_id}` with 24h TTL. It stores the user
  message and final assistant answer only, not tool_use/tool_result blocks.
- Budget enforcement is DB-backed in `api.services.budget`: sum
  `llm_extractions.input+output` plus `llm_query_dispatches.total_input+total_output`
  since local midnight. Cache-read tokens are excluded. Default pre-call
  buffer is 2000 tokens.

### Telegram delivery

- `bot/handlers.py::on_text` wraps message processing in
  `typing_action()`, which sends `send_chat_action("typing")` immediately
  and refreshes every 4s until dispatch finishes or errors.
- `bot/delivery_send.py` is the shared delivery helper:
  `render_chunks(text)` is pure sanitize+split, and `send_chunked(...)`
  sends chunks sequentially with `ParseMode.HTML`. Inline buttons attach
  only to the last chunk.
- Query errors are mapped by `app.queries.delivery.handle_query_error()`.
  User-facing errors are short and sent as one HTML message.
- `/clear` clears only query history via `app.queries.history.clear_history`.
  It does not touch pending writes, clarification state, pairing, or rate
  limit keys. Reply is always `Listo, contexto limpio.`

### Query endpoint

`POST /api/v1/queries/test` exists for curl/debugging without Telegram.
Auth uses `current_user` (`X-Shortcut-Token` preferred, `X-User-Id` dev

### The three nudge types

- **missing_income** ‚Äî user has ‚â•5 transactions in last 7d AND zero
  `amount > 0` transactions in last 30d. Dedup key
  `missing_income:{user_id}:{YYYY-MM}` (one per calendar month). Priority
  always `normal`.
- **stale_pending_confirmation** ‚Äî a `pending_confirmations` row has
  `resolved_at IS NULL AND created_at < now - 48h`. Dedup key
  `stale_pending:{pending_confirmation_id}`. Priority `normal`. The
  dispatcher (bot/pipeline.py) writes this table in parallel to Redis
  at propose-time; Redis is still the 5-minute session truth, Postgres
  is the 48h+ audit.
- **upcoming_bill** ‚Äî a `notification_events` row with `status='pending'`
  whose `payload_snapshot.due_date` (or `event_date` for custom_events)
  falls within the next 72h. Dedup key `upcoming_bill:{notification_event_id}`.
  Priority `high` when due_date ‚â§ today+1 (inside 24h on the calendar-date
  model); `normal` otherwise.

### Four anti-saturation rules (hard-coded, not user-configurable)

All constants live in `api/services/nudges/policy.py` as named module-level
values ‚Äî no magic numbers elsewhere.

1. **GLOBAL RATE LIMIT** ‚Äî 1 delivered nudge per user per 48h. HIGH
   priority bypasses this rule (it also doesn't count against future
   rate limits). Implemented as: check prior `user_nudges.status='sent'
   AND priority='normal' AND sent_at >= now-48h`; plus a same-run flag
   that flips after the first normal send so a single worker pass never
   sends two normals.
2. **PER-TYPE SILENCING** ‚Äî 2 dismissals of the same nudge_type within
   the last 30d silence that type for that user for 14d. Silence rows
   live in `user_nudge_silences` and are inserted at the moment the
   threshold is crossed (not computed live on every evaluator pass).
   A duplicate silence row is NOT inserted while an earlier one is
   still active.
3. **QUIET HOURS** ‚Äî 21:00 ‚Üí 07:00 in the user's timezone
   (`users.timezone`, default America/Costa_Rica). Nudges generated in
   the quiet window stay `status='pending'`; the next delivery pass
   outside the window sends them. Applies to HIGH too ‚Äî no emergency
   delivery at 3am.
4. **DEDUP** ‚Äî `UNIQUE(user_id, dedup_key)` on `user_nudges` + the
   orchestrator's `INSERT ... ON CONFLICT DO NOTHING` make re-runs of
   the evaluator idempotent by construction.

### Schema (migration 0008)

- **user_nudges** ‚Äî `id`, `user_id` FK `ON DELETE CASCADE`, `nudge_type`
  (CHECK `missing_income|stale_pending_confirmation|upcoming_bill`),
  `priority` (CHECK `normal|high`), `dedup_key` (UNIQUE with user_id),
  `payload` JSONB, `source_notification_event_id` FK `ON DELETE SET NULL`,
  `status` (CHECK `pending|sent|dismissed|acted_on|expired|suppressed`),

  the delivery worker; `(user_id, status, created_at DESC)` drives listing.
- **user_nudge_silences** ‚Äî `id`, `user_id`, `nudge_type`, `silenced_until`,

## Architecture Principles

1. **Data before AI** ‚Äî Conversational features are downstream of accurate data pipelines. No LLM layer until Phases 0‚Äì4 are solid.
2. **Deterministic pushback** ‚Äî The affordability engine uses math, not LLM generation. The LLM wraps results in natural language. The LLM never calculates.
3. **Personal before product** ‚Äî Use the system yourself for 4+ weeks before adding multi-tenancy. Watch where it breaks.
4. **No premature complexity** ‚Äî Rule-based categorization before ML. Single currency before multi-currency. Telegram before WhatsApp. API LLM before self-hosted.
5. **Every phase ships something usable** ‚Äî Each phase produces a feature you can use that day.

## The Pushback Engine (Phase 6 ‚Äî Critical Design)

This is the hardest feature. The engine is **deterministic**, not LLM-generated.

```python
def assess_affordability(monthly_income, fixed_expenses, existing_commitments, desired_amount, desired_timeline_months):
    monthly_disposable = monthly_income - fixed_expenses - existing_commitments
    monthly_needed = desired_amount / desired_timeline_months
    feasible = monthly_needed <= monthly_disposable * 0.8  # 80% safety margin

    return {
        "feasible": feasible,
        "monthly_disposable": monthly_disposable,
        "monthly_needed": monthly_needed,
        "shortfall": max(0, monthly_needed - monthly_disposable),
        "suggestion": generate_suggestion(...)
    }
```

The LLM's system prompt enforces this separation:
- NEVER invent numbers. Always call a tool to get real data.
- When asked "can I afford X", call the affordability tool and report its findings honestly.
- If not feasible, say so clearly and offer alternatives (extend timeline, reduce target).
- Don't sugarcoat. Don't be harsh. Be direct and constructive.

## Technical Debt (tracked, not blocking)

- **No real scheduler yet.** The three Phase 4 batch jobs (`generate-occurrences`, `mark-overdue`, `compute-notifications`) are exposed as `POST /api/v1/jobs/*` protected by `X-Shortcut-Token`. Phase 5+ will wire them to a proper scheduler (likely `arq` or Celery on top of Redis). Until then they must be triggered by an external cron or the iPhone Shortcut.
- **No `POST /jobs/*` idempotency key.** Safe to re-run because each underlying operation is idempotent, but we don't throttle concurrent calls.
- **Legacy `events` table removed in migration 0005** (was a Phase 1 stub that was never wired up). The new Phase 4 entity is `custom_events`.
- **`recurring_bills.linked_loan_id` FKs to `debts.id`** (there is no separate `loans` table; `debts` holds installment loans). Mortgages and personal loans should reuse this column.
- **RRULE validation.** We accept any iCalendar RRULE string via `recurrence_rule` but don't validate it until generation time. A bad rule will surface as an exception during `generate_occurrences`, not at write time.
- ~~**No per-user scoping on Phase 4 tables.**~~ Resolved in Phase 5a ‚Äî every Phase 4 table now carries a `user_id` FK with `ON DELETE RESTRICT` and per-user composite indexes (see migration 0006).
- **`X-User-Id` dev shim is shipped to production code.** Tracked here so we don't forget to remove it. See the Phase 5a section below for details.
- **Phase 5d: no `pending_confirmations` retention policy.** One row per bot proposal; ~tens of inserts/day in personal use, no pruning. At 10k rows/year this is fine for Postgres. When it crosses a threshold, add a nightly `DELETE WHERE resolved_at < now() - '1 year'::interval`.
- **Phase 5d: "M√°s tarde" / "Recordame ma√±ana" don't actually schedule a deferred re-nudge.** They mark the nudge `dismissed` and count toward silence like any other dismiss. The reply text is honest about it. A proper deferral mechanism needs a scheduler (see first bullet) ‚Äî wire it when the scheduler lands.
- **Phase 5d: `upcoming_bill` can't re-nudge while status=dismissed.** The `UNIQUE(user_id, dedup_key)` means once the user "descarta" a bill reminder, the next evaluator run won't re-generate it ‚Äî even if the bill is still within 72h. By design: the user said "no" and we respect it.
- **Phase 5d: one `upcoming_bill` candidate per pending `notification_events` row.** Phase 4's `compute_pending_notifications` creates one row per `(bill_occurrence, advance_days)` pair ‚Äî so a bill with `advance_days=[7,3,1,0]` produces 4 pending rows once due_date is within 72h, and the evaluator surfaces all 4 as distinct nudges (distinct dedup_keys). The 48h rate limit caps delivery to ~1 per window in practice, but this is still more fan-out than ideal. A future fix collapses multi-tier notifications per bill into one nudge before insert.
- **Phase 6a: splitter is wired but mostly no-op in normal queries.** Telegram delivery now always runs `sanitize_telegram_html ‚Üí split_for_telegram ‚Üí sequential sends`, but `app/queries/llm_client` still uses a low `max_tokens` response cap, so real multi-chunk sends are rare. Before raising output limits substantially, keep or add an e2e that produces a >3900-char real response and verifies every chunk is valid Telegram HTML.

- **`X-Shortcut-Token`** (real, server-resolved): identifies the caller via `users.shortcut_token`. Required by `POST /transactions/shortcut` and every `POST /jobs/*`.

- Rotation invalidates the previous token instantly (next request with the old value gets 401).
- Used as the iPhone Shortcut auth header and as the authenticated API token for user-scoped endpoints, including Telegram pairing and `/api/v1/queries/test`. Telegram message delivery itself resolves users by `users.telegram_user_id`.

4. After 0006 is applied, the three `LEGACY_*` env vars are unread by the running app and can be removed from `.env` (the runtime no longer references `DEFAULT_USER_ID` or `SHORTCUT_TOKEN` either; both were removed from `api/config.py`).
